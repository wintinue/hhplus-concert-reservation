### 메시지큐를 사용하는 이유를 상기해봅시다

- 서비스간의 관심사 분리
    - 책임 범위 최소화 (결합도 감소)
    - 장애 전파의 최소화
- 분산 메시징 시스템이라면 대규모 데이터 처리 가능
    - 수용 가능한 데이터 처리량을 Scale-out 할 수 있음
- 시스템 전체의 안정성 강화
    - 처리에 실패한 메시지를 재처리할 수 있음

### Kafka

- 대용량 환경을 위해 디자인된 분산 메시징 시스템
    - 파티션이라는 개념을 통해 분산 처리를 지원하여 다량의 메시지 읽기/쓰기를 지원
    - 이러한 특성으로 인하여 기본적으로 각 메세지의 처리 순서를 보장하지 못함
        - 키가 같은 메시지라면 순서가 보장되지만, 이 경우 부하 분산은 이루어지지 않는다는 사실을 기억해야한다!
- 물리적인 인스턴스(브로커) 스케일링에 대해 자유로우나, 논리적으로 파티션을 자유롭게 축소하는 것은 어려움
- 오프셋과 커밋 개념을 통해 과거에 발행된 메시지를 나중에 다시 읽을 수 있음
    - At Least Once(최소 한번 발송)를 보장하지만, 메시지를 중복 컨슘 할 가능성이 있으므로 이를 고려해야 함

### Kafka 이외에 다른 메시지 큐

- Redis Pub/Sub
    - 엄밀히 따지면 그냥 서비스간의 통신을 위한 이벤트 버스 (메시지큐라고 보기는 어려움)
        - 메시지 발행 시점에 컨슈머가 없으면 그대로 메시지는 소실됨
    - 매우 빠른 속도로 동작
    - 기존에 있는 Redis 인프라를 그대로 활용할 수 있음
    - 확장성에 한계가 있음
- RabbitMQ
    - AMQP 구현체 (표준 인터페이스가 있고, 지원하는 기능이 다양함)
    - 기본적으로 컨슈머 그룹 개념이 없음
        - 하나의 메시지를 여러 서비스가 받는 것이 어려움
    - Kafka에 비해서는 확장성이 떨어짐
- AWS SQS
    - 매우 단순한 사용법으로 러닝커브가 거의 없음
    - 지원 기능 빈약
    - 서버리스 서비스이기 때문에 확장이 매우 용이
    - 벤더 종속적 (AWS 이외의 인프라 환경에서 사용 어려움)
- AWS Kinesis
    - Kafka와 비슷한 구성의 사용법
        - 대용량 데이터 처리에 효율적으로 동작함
    - 서버리스 서비스이기 때문에 확장이 매우 용이
    - 벤더 종속적 (AWS 이외의 인프라 환경에서 사용 어려움)

### 실제 현업에서 트랜잭션을 대하는 우리의 자세

> 대용량 트래픽을 제어하기 위한 3번째 방법, 트랜잭션의 범위를 적절하게 재설계하는 방법 및 전략 등에 대해 알아보았습니다.
> 
- 결국 대용량 트래픽을 적절히 제어하기 위해서는 더 많은 요청을 수용할 수 있도록 예외의 잘못된 전파 등을 방지하는 것이 중요
    - 각 요청에 대한 빠른 처리를 통한 자원 반환: API 요청 자원, DB Connection 등
    - 적절한 책임 분리
- 더 많은 트래픽을 무리없이 수용할 수 있다는 것은 “각 트래픽을 빠르고 성공적으로 처리할 수 있다”는 전제가 달성되어야 함
    - 트랜잭션의 범위를 적절히 제어
    - 예외(예측이 어려운 오류)가 트랜잭션 전체 실패로까지 이어지는 상황을 최대한 제어할 수 있어야 함
- 적절한 트랜잭션 범위를 설정하기 위해 생각해야할 요소
    - 너무 긴 작업을 수행해 DB Connection 고갈로 이어지지 않아야 함
    - 트랜잭션과 연관 없는 작업들에 의해 주요 비즈니스 로직이 영향받지 않아야 함

### Case Study

- **유저 회원가입 CASE**
    
    ```kotlin
    Tx {
    	가입정보 검증()
    	회원정보 발급()
    	환영메일 발송()
      웰컴쿠폰 지급()
    }
    ```

    - 생각할 거리: 중복유저를 검사하는 방식
        - 본인인증
        - DB에서 이메일 정도만 검증하는 방식
- 위와 같은 구성으로 트랜잭션을 구성했을 경우 발생할 수 있는 문제는 무엇일까?
    - 메일 발송 관련 이슈
        - 환영메일 발송이 네트워크 문제로 오래 걸림 → DB Connection 지연
        - 환영메일 발송이 실패 → 회원가입 실패
    - 웰컴 쿠폰 발급 관련 이슈
        - 데이터 저장 실패 → 회원가입 실패
- 우리의 핵심 비즈니스 로직 = 회원가입
    - 현재 트랜잭션에서 하는 일: 회원가입, 환영 메일 발송, 웰컴 쿠폰 발급
    - 우리가 배운 방식대로 다시 정리해보면?
        - 주요 로직: 회원가입
        - 부가 로직: 환영 메일 발송, 웰컴 쿠폰 발급
    
    주요 로직이 실패하였을 경우엔 해당 회원이 서비스에 가입되지 못 했기 때문에, 환영메일을 발송해서도, 웰컴쿠폰을 지급할 수도 없습니다. 즉, 주요 로직의 결과는 부가 로직의 수행 여부에 영향을 끼치지만 부가 로직의 결과가 주요 로직의 수행시간 증가나 실패로 이어져서는 안됩니다. 
    
     하지만 어떠한 이유에서 환영메일을 발송하지 못 했다고 했을 때, 회원가입이 실패하는 것은 매우 이상한 일입니다. 나아가 메일 발송 API 가 지연되어 전체 회원가입 로직을 처리하는 트랜잭션이 오래 대기하게 되면 문제가 발생할 수 있습니다. 차라리, 어떻게든 환영메일을 발송하고 싶다면 해당 회원에 대한 발송 여부를 저장하고 주기적으로 해당 데이터를 확인해 발송하지 않은 회원에게 환영메일을 발송해 주는 것이 적합한 설계입니다. 
    
     또한 웰컴쿠폰 지급의 경우도 같습니다. 웰컴쿠폰이 반드시 지급되어야 하는가 ? 지금 당장 쿠폰 지급이 실패하더라도 회원가입이 실패하면 안됩니다. 쿠폰 시스템이 별개일 때는 오히려 KAFKA 메세지 같은 매개체를 이용해 언젠가 지급되기만 한다면 된다는 정책을 추가할 수도 있습니다.
    
    ```java
    Tx {
    	가입정보 검증()
    	회원정보 발급()
    	가입완료 이벤트 발행()
    }
    
    TxAfter(가입완료 이벤트) {
    	이벤트 수행 상태 데이터 저장()
    }
    
    EventListen1(가입완료 이벤트) {
    	환영메일 발송()
    	발송 성공 상태 변경()
    }
    
    EventListen2(가입완료 이벤트) {
    	웰컴쿠폰 발급()
    	발급 성공 상태 변경()
    }
    ```
    
    위와 같이 설계를 변경함으로서 각 비즈니스 로직 ( 메인 / 부가 ) 는 본인의 관심사만 온전히 처리할 수 있게 되었고, 적절한 영향도에 따른 위상을 갖추게 되었습니다. 이러한 비즈니스 로직의 설계적 관점을 여러분들이 맞닥드리는 다양한 문제에 잘 고려해보고 올바른 트랜잭션 관리로 이어질 수 있도록 노력해보시기 바랍니다 😄

## **🎯들어가면서, Why 장애대응?**

![슬라이드3.PNG](Step%2010%20Incident%20Management%20Lifecycle/%EC%8A%AC%EB%9D%BC%EC%9D%B4%EB%93%9C3.png)

- 장애대응에서 실제로 가장 중요한 포인트
    
    **장애가 아예 발생하지 않는 소프트웨어는 없습니다.** 우리가 고민해야 할 것은 ..
    
    1. 예방 가능한 장애를 방지할 것
    2. 장애가 발생했다면 빠르게 전파하고 해결할 것
    3. 발생했던 장애는 다시는 발생하지 않도록 할 것
    
    실제로 모든 장애는 우리가 예측 가능한 범위 밖에서 발생하므로 “장애”라고 부르는 것임을 기억합시다.
    
    모니터링 시스템 등을 통해 장애 발생을 빠르게 탐지하고 전파할 수 있는 환경을 갖추고, 실제 장애가 발생했을 때 해당 상황을 테스트 케이스로 작성해 우리 코드 내에서 예측 가능한 범위 내의 문제로 가져오는 방법을 통해 장애 상황에도 견고한 어플리케이션을 만들고, 유지하는 데에 힘쓰는 방법을 알아봅니다.
    
    <aside>
    💡 **주의깊게 볼 문제상황**
    1. 데드락, 정합성 문제가 발생하는 API
    2. 서킷브레이커가 동작해야하는 API
    3. SQL 연산 강도가 높은 API
    
    </aside>
    
    - `Logging` - 성능적 우려가 되는 포인트들은 사전에 개발과정에서 인지할 수 있습니다. 인지된 사항들에 대해 로깅을 통해 장애가 실제 발생하기 이전에 문제 상황을 인지하고 개선한다면 장애로 이어질 확률을 낮출 수 있습니다.
    
- 체계화된 장애 대응 프로세스
    
    실제로 장애 상황이 발생 했을 때에는 매우 당황스럽고 혼란스러우므로 침착함을 유지하며 이 문제를 해결해나갈 방법을 생각하기 쉽지 않습니다.
    
    따라서 장애가 발생했을 때 기계적으로 대응 할 수 있는 매뉴얼을 만들어놓는 것이 좋습니다.
    
    장애 상황이 발생했을 때, 어떤 프로세스를 통해 장애상황을 전파하고 복구하는지 장애대응 프로세스를 만들어봅시다.
    
    이를 기반으로 우리 회사 내의 현재 장애 대응 프로세스와 함께 생각해보며, 더 나은 점은 무엇인지 더 개선해야할 점은 무엇일지 등을 함께 고민해보면 더 좋은 프로세스를 만들어 낼 수 있을 거라고 생각합니다.

### 모니터링 및 로깅 중인 지표

1. 서버의 리소스 ( Latency, Disk ) 등의 지표를 확인하고 있습니다.
    - MSA 환경으로 서비스를 구성 중에 있고, 각 서비스는 상호 필요한 정보를 내부 API 통신으로 요청합니다.
    Latency를 모니터링 하여 의존적인 시스템의 장애가 전파되지 않도록 합니다.
    - 내부 서비스 및 외부 통신에 대한 정보들을 최대한 상세히 모니터링 하고 있습니다.
        - MSA 환경에서 가장 중요한 요소중 하나 = Observability
            - 리퀘스트의 흐름: A 서비스 → B 서비스 → DB
        - 이에 따라 Disk 용량 부족 등의 문제가 발생할 수 있습니다.
2. 데이터를 올바르게 파싱하지 못하는 경우에도 로깅 및 알림을 받고 있습니다.
    - 다른 서비스의 스펙이 변경되거나, 응답을 변환하는 과정에서 문제가 발생한 경우 이를 대응해 시스템이 유기적으로 정상 동작할 수 있도록 개선하기 위함입니다.
3. 에러의 빈도가 잦은 경우에도 알림을 받고 있습니다.
    - 의도하지 않은 비즈니스 로직으로 인해 발생했을 가능성이 있습니다.
    - 의도한 에러였지만 문의가 발생할 가능성이 있습니다.
4. 모든 Request, Response를 로깅합니다.
    - MSA 환경에서 운영되는 서비스 특성상 중간에서 Trace 가 끊기는 경우 데이터의 흐름을 올바르게 파악할 수 없는 문제가 생깁니다.

가장 대중적인 검색 엔진 서비스입니다. 로그를 보다 효율적이고 빠르게 검색하기 위해서 사용합니다.
흔히 ELK 스택이라고 불리는 구성에서 가장 핵심이 되는 컴포넌트입니다.

Datadog Slack Application

- Datadog 설정한 지표들을 Slack으로 받고 있습니다. 에러의 빠른 탐지를 위해서 사용합니다.

### 각종 지표를 통합한 DashBoard

Grafana를 통해 장애 및 API 별 요청수, 트래픽 처리효율, 캐시 메모리 등 다양한 지표를 파악할 수 있습니다.

또한 각종 서비스의 에러율 ( Error / Request ) 을 시각화하여 확인할 수 있도록 합니다.

웹으로 제공될 뿐 아니라 엔지니어들이 시각적인 지표로 쉽게 확인할 수 있다는 장점이 있습니다. 연관된 서비스에서 문제가 발생하더라도 해당 서비스의 지표를 파악하여 Root Cause 를 쉽게 파악할 수 있습니다.
또한 로그가 아닌 시각화 된 정보이기 때문에, 다른직군의 사람들도 보다 쉽게 문제를 파악할 수 있습니다.

### **부하 테스트**

![슬라이드12.PNG](Step%2010%20Incident%20Management%20Lifecycle/%EC%8A%AC%EB%9D%BC%EC%9D%B4%EB%93%9C12.png)

![Untitled](Step%2010%20Incident%20Management%20Lifecycle/Untitled%202.png)

부하 테스트는 진행 시간과 트래픽 양에 따라 약 4가지 포맷으로 분류합니다. 일반적으로는 짧은 시간단위의 **Load Test** 및 폭발적인 트래픽에 대한 내성을 점검할 **Peak Test** 에 우선순위를 가져갑니다.

**Load Test ( 부하 테스트 )**

- 시스템이 예상되는 부하를 정상적으로 처리할 수 있는지 평가
- 특정한 부하를 제한된 시간 동안 제공해 이상이 없는지 파악
- 목표치를 설정해 적정한 Application 배포 Spec 또한 고려해 볼 수 있음

**Endurance Test ( 내구성 테스트 )**

- 시스템이 장기간 동안 안정적으로 운영될 수 있는지 평가
- 특정한 부하를 장기간 동안 제공했을 때, 발생하는 문제가 있는지 파악
- 장기적으로 Application 을 운영할 때 발생할 수 있는 숨겨진 문제를 파악해 볼 수 있음 ( feat. Memory Leak, Slow Query 등 )

**Stress Test ( 스트레스 테스트 )**

- 시스템이 지속적으로 증가하는 부하를 얼마나 잘 처리할 수 있는지 평가
- 점진적으로 부하를 증가시켰을 때, 발생하는 문제가 있는지 파악
- 장기적으로 Application 을 운영하기 위한 Spec 및 확장성과 장기적인 운영 계획을 파악해 볼 수 있음

**Peak Test ( 최고 부하 테스트 )**

- 시스템에 일시적으로 많은 부하가 가해졌을 때, 잘 처리하는지 평가
- 목표치로 설정한 임계 부하를 일순간에 제공했을 때, 정상적으로 처리해내는지 파악
- 선착순 이벤트 등을 준비하면서 정상적으로 서비스를 제공할 수 있을지 파악해 볼 수 있음

장애는 자연재해와 같습니다. 아무리 예방해도, 예상치를 뛰어넘으면 문제가 생기죠.

결국 사람은 자연재해를 완벽하게 막을 수 없습니다. 다만 예방에 더 힘을 쓸 수 있습니다.

- p.s. [자연재해 감지기](https://downdetector.com/)

이때 중요한건 `심리적 안정감` 입니다.

로켓 발사는 수없는 실패를 겪습니다. 만약 그럴때마다 실패한 엔지니어를 자른다면? 누가 로켓을 우주로 보낼 수 있을까요? 사람은 실패를 통해 배운다는 사실을 기억해주셨으면 합니다.

실패했을 때 원인을 분석하여 다시 같은 문제가 벌어졌을 때를 대비해야합니다.

문제를 일으킨 코드를 만든 사람을 색출하고, 그에게 책임을 지우면? → 아무도 도전적인 업무를 하지 않습니다. 커뮤니케이션이 완전히 경색됩니다.

이런 조직은 백날 장애대응 훈련을 해도 무용지물입니다.

그러니 여러분이 만약 어디선가 장애 대응을 해야한다면! 프로세스를 점검하는 것도 중요하지만, 조직내에 심리적인 안정감을 끌어올리는것도 매우 중요하다는 사실을 알아주시면 좋겠습니다.

### **`필수과제 - 기본 부하테스트 진행`**

![슬라이드20.PNG](Step%2010%20Incident%20Management%20Lifecycle/%EC%8A%AC%EB%9D%BC%EC%9D%B4%EB%93%9C20.png)

- 부하 테스트 대상 선정 및 목적, 시나리오 등의 계획을 세우고 이를 문서로 작성
    - 어떤 부분을 부하 테스트를 해야할까? → 왜 이 부분을 부하 테스트를 해야할까?
    - 어떻게 부하 테스트를 진행할까?
    - 부하 테스트를 진행하기 위한 인프라와 시간을 얻어내야겠다는 마인드로 직장 상사에게 보고한다고 생각하며 작성해보아요!
- 적합한 테스트 스크립트를 작성하고 수행
    - 실제 작성한 테스트 스크립트와 수행 결과를 정리하여 제출해주세요.
    - Application에 할당하는 자원(e.g., 도커의 CPU/Memory 제한)을 조절해보면서 이 앱의 적절한 배포스펙이 어떤 지점인지도 찾아보면 BEST!

### **`선택과제 - 장애 대응 매뉴얼 작성`**

![슬라이드21.PNG](Step%2010%20Incident%20Management%20Lifecycle/%EC%8A%AC%EB%9D%BC%EC%9D%B4%EB%93%9C21.png)

- 시스템 내의 병목을 탐색 및 개선해보고 ****장애 대응 매뉴얼을 작성하고 제출
    - 부하 테스트를 진행하며 얻은 데이터를 활용하면 좋습니다.
    - 완벽히 개선이 어려운 포인트에 대한 대응 매뉴얼과 아예 예측을 못한 포인트에 대한 대응 매뉴얼을 모두 만들면 BEST!
- 최종 발표 자료 작성 및 제출
    - 어떤 내용들이 들어가면 좋을까?
        - 우리 시스템의 전반적인 구조
        - 우리 서비스는 어느정도까지의 요청을 수용할 수 있는가 (주요 기능별)
        - 어떤 부분에 취약점이 있는가
        - 장애가 발생하는 경우 어떻게 대응하면 되는가
    - 우리 동료들 뿐 아니라 우리 서비스의 모든 이해 관계자들에게 발표한다고 생각해봅시다!
